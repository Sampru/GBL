Detailed Description

ITF: InTerFace definitions and transcoding.

Header file
itf_def.h
Library
Eng

Interface related definitions and transcoding

  • Engines
  • The main philosophy of the interface engines is that all the interface components should be described in data files. This gives flexibility to the development and it enables a quick analysis of each part.
  • The desired behaviours a characterised and once they are correctly implemented the interface is uniformised.
  • Having the information in descriptor files has several advantages against storing it in the database:
  • We have a trazable historic of the modification (SCCS)
  • It's much faster loading in remote executions
  • We don't have replicated information in each database
  • We don't need to create complex scripts to update the information
  • ...
  • Descriptor files (*.dat) and descriptor editor (deda)
  • Each interface engine has one or more associated descriptors.
  • For each descriptor we'll normally have a "schema" file which describes its structure. This is mainly done to enable visualising and modifying this information meaningfully with our descriptor editor, deda.
  • Descriptor files' path: environment variable "DATPATH"
  • There are several reasons that make convenient having several descriptor directories. Firstly, when we want to change a descriptor we typically have a local (modified) version coexisting with the one in the reference directory. Additionally, if we have different libraries using the same philosopy (application dependent, e.g. Fox, generic, e.g. engines, ...) we'll want to have their files in different locations, as their actualisation will normally be asynchronous.
  • For doing so we define an environment variable which specifies a list of directories which will be visited in sequence. When a file is searched for the first match will be taken.
  • When several files are searched for (e.g. error codes) only the first file with the same name will be chosen (should there be several in different locations).
  • Multi-language
  • For any descriptor column susceptible to be made multi-language (window title, criterion label, field label, ...) we may have support for other languages. This will be carried out by specifying a global variable with the language (ESP, FRA, GBR, ...) and defining an equivalent field for each supported language. For example, a title field "tit" may be specified in Spanish by defining a "titESP" column in the appropriate schema and translating the titles in the "titESP" column of the data file.
  • Should the contents of a language-specific field be empty the default contents will be used.
  • See ITFnormaliseLanguage () here for more details.

Gets the graphic resources set for the specified set.

Gets the graphic resources set for the specified set. Actually the graphic objects that the engines manages are:

  • winTit: Title of the window
  • mtxTit: Title of the matrix
  • mtxDta: Cells of the matrix
  • butLab: Buttons labels
  • butTrtLab: Treatment button label
  • fldLab: Window fields label
  • fldDta: Window fields datas
  • untLab: Unit label
  • mnuBar: Menu bar
  • staBar: Status bar
  • scrolling: Scrollingin popup objects By the moment the engine manage only the fonts.
  • The fonts associated to each graphic resource set are explicited in deda.doc document. The font and colors of all other objects are fixed. ATTENTION !! Actualise the number of graphic resources ("NbGrs" variable) each time that we add a new resource.
Parameters
grsGraphic resources set
LdGrs(OUT). List of resources
Returns
Error code (0: OK)
