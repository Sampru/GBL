Detailed Description

Error management.

Header file
errDef.h

This module provides a framework to organise errors by type/code.

  • - The type/code couple results in an entry to an error message list, similar to that of "errno" in the standard C management (see strerror ()).
  • - The error type may be generic (e.g. memory, file problems) or error types associated with a treatment (one type by treatment).
  • - Each error has an error message and a description. The error message (associated with the type/code) contains the generic error (e.g. "No such file or directory") and the description specifies the details on the error (e.g. "my_filename").
  • - One or various files are suposed to have the list of error messages for every defined couple type/code, e.g. "No such file or directory".
  • - These files centralise error types and codes so that no recompilation of all programs using it is required for each new line added.
  • - Each treatment may use local macrodefinitions initialised appropriately for types/codes to make code readable (e.g. GENERIC, FILE_NO_EXIST).
  • - These message files are searched for in the directories pointed by the environment variable "DATPATH".
  • If any file cannot be read no message will be provided (but the type/code will be treated anyway).
  • If the same file is read on various directories, it get the first file, that is, it get the error files in the order that the directories are into the environment variable "DATPATH".
  • - The typical use of this module may be described as follows:
  • - 0) Get the list of error messages and set it as the default list of messages, that is, local list for the error module.
  • Normally, we must get the list of the error files and then get the error messages of these files.
  • Whethever, we can set the error messages list as the default error messages by other method.
  • - ITFgetErrorFiles (&LdErr); * Get the list of the error files *
  • - ERRsetFilesList (LdErr); * Save the list of the error files *
  • - ERRgetMsgList (LdErr, &LdMsg); * Get the error messages (generic behaviour) *
  • - ERRinitMsgList (LdMsg); * Set the error messages as a local list *
  • - The error messages list is initialised from the beginning, it not expects to go to read the first error message.
  • - 1) Remove the list of previous errors
  • ERRresetError ();
  • - 2) When an error occurs add it to the error list:
  • + ERRsetError (MY_TRT, PRICE_MISSING, Contract);
  • If we want to write the GB errors in the error message list we will use:
  • + ERRsetGbError (Ctx);
  • And for the DYN error messages:
  • + ERRsetDynError (Ld);
  • - Depending on the treatment we may decide to finish it or to continue, so that all errors are scanned for.
  • - Alternatively, when we want to add a list of error messages we may use a list:
  • ERRsetErrorLst (MY_TRT, MATRIX_MISSING, "pm_mtx", Ld);
  • - 3) We have also a possibility to put messages as warnings or informations like the error system
  • ERRsetWarn (MY_TRT, PRICE_MISSING, Contract); –> warnings
  • ERRsetInfo (MY_TRT, PRICE_MISSING, Contract); –> informations
  • - When we want to add a list of warning/information messages we may
  • ERRsetWarnLst (MY_TRT, MATRIX_MISSING, "pm_mtx", Ld);
  • ERRsetInfoLst (MY_TRT, MATRIX_MISSING, "pm_mtx", Ld);
  • - The principal difference between the error messages level is the concept they are used for. Here are the rules to use each level in some situations:
  • + Error :
  • A error in the code is produced. The user would have to call to the application suppervisor. The dialog window colour is red.
  • + Warning :
  • It is a message to tell the user that there is something missing or incorrect. It is the user who can correct the problem. The dialog window colour is blue.
  • + Information :
  • Message shown to the user to let him know about something. The dialog window colour is wheat.
  • - 4) When the treatment is finished, check if any error or warning has occurred:
  • if (ERRisThereAnyError ())
  • ...
  • It is possible to verify if there is an error in the GB and DYN modules with:
  • if (ERRisThereAnyErrorGb (Ctx))
  • ...
  • if (ERRisThereAnyErrorDyn (Ld))
  • ...
  • This check does not stop the execution of the code if there is an information and if we want to check if there is also any information:
  • if (ERRisThereAnyMsg ())
  • ...
  • - 5) A copy of the error list may be obtained through the function:
  • ERRgetErrorList (&LdErrors);
  • This list contains four fields:
  • typ: type of error (int)
  • cod: Error code (int)
  • msg: Error message (ERR_MSG_TYPE, ERR_MSG_LENGTH, char *)
  • des: Error description (ERR_DES_TYPE, ERR_DES_LENGTH, char *)
  • Alternatively, an already formatted list may be obtained:
  • ERRgetFmtList (&LdErrorsFmt);
  • This list contains one field:
  • fln: Error line (ERR_FLN_TYPE, ERR_FLN_LENGTH, char *)
  • - 6) Three static lists are mantained: one to hold the list of messages (which is loaded once forever the first time that is required), another one to hold the list of errors accumulated and the third one to hold the list of the read error files. Additionally, the filename initialised for the message file must also be freed.
  • At the end of the application (or at any moment, as they will be reinitialised) this memory should be freed through the destructor:
  • ERRdestroy ();
  • The structure of the static list with the read error files lcERRfilesLst is:
  • file: Name of the file
  • dirname: Directory
  • The range of values for the different error codes files are:
  • + uti_c: 1000 - 1999 (err_cod_uti.dat)
  • + egut: 2000 - 2999 (err_cod_egut.dat)
  • + eng: 3000 - 3999 (err_cod_eng.dat)
  • + prr: 4000 - 4099 (err_cod_prr.dat)
  • + bat: 4100 - 4199 (err_cod_bat.dat)
  • + aluSql: 4200 - 4299 (err_cod_sql.dat)
  • + app: 10000 - 19999 (err_cod_xxx.dat)
  • When an error occurs, to stop the execution, use the macro-definition named ERRdef. Also, it is neccessary to define a ERRend label to go when an error occurs.
  • To verify is there is an error in a function, there are some macro-definitions:
  • - ERRchk –> When an erro occurs, it goes to the ERRend label on the code
  • - ERRchkGb –> Same as ERRchk but only for GB functions
  • - ERRchkErr –> Same as ERRchk but it allows to put a error message
  • - ERRchkNoTrace –> Same as ERRchk but sends no trace to the log file
  • - ERRchkDyn –> Same as ERRchk but only for DYN functions
  • These macro-definitions NEVER mustn't be used after the ERRend label. ¡¡¡ NEVER !!!
  • All these functions write the returnd value of a function in ErrCd and compare it with 0; if it is the same, there is no error.
  • - Multi-language
  • To be made multi-language the error messages we may have support for other languages. This will be carried out by specifying a global variable with the language (ESP, FRA, GBR, ...) and defining an equivalent field for each supported language.
  • Should the contents of a language-specific field be empty the default contents will be used.

static int lcERRnorLan ( void *  LdMsg)
static

Normalise the error list in terms of language.

Normalise the error list in terms of language. Take the language associated columns and replace its contents by those of the default language. When the application instance has defined a Language to load labels it is taken into account, but when there is not any language defined the engine's default, so English, label are used. When using a specific language different than the default one, if there is not defined any label for this language the default one will be got.

Global variables
  • Language: Language used to load labels. If it is NULL it is not taken into account. If it is empty english is taken in to account.

LdMsgMessage list
Returns
