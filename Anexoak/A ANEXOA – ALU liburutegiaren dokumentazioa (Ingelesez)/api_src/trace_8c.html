Detailed Description

Dinamic traces management module.

Header file
trace.h
Acronyme
FMT TR
Library
cross

Dinamic traces management module

  • It generates a context (global variable) with:
  • The known vocabulary of components (keywords + acronymes) got from an external file
  • - A initialisation flag
  • - A trace activation flag
  • - A table to allow kowing the component calls (Usually trace.dat in TRACE_FILE environment variable)
  • - A table for trace orders. (Usually order_trace.dat in ORDER_FILE environment variable)
  • - The quick search of the index of a keyword or acronym is made with the function "bsearch" which needs a previous sort of the tables with "qsort"
  • - Each time the component is called by a TRtrace with the keyword "ENTREE", the table of calls is incremented for the acronyme passed to TRtrace.
  • - The same tabel is decremented with "SORTIE".
  • - This way, we may easyly know when a trace is activated with a call condition, if this condition is realised or not. – Example:
    • TRinit => Parsing of TRACE_FILE file
    • Load of acronym table
    • Load of keywords table
    • Qsort of these tables.
    • TRsetTrace("XX","YY","key") => Means that we want to trace "YY" acronym or module only with "key" keyword, and when this module has been called from "XX" acroynym module.
  • => bsearch of XX in ix
  • => bsearch of YY in iy
  • => bsearch of key in ik.
  • orders[iy][ik][ix] = 1;
    • On XX module we'll have:
    • TRtrace("XX","ENTREE",...)
    • => bsearch de XX vaut ix
    • => appel[ix] est incremente.
    • Then XX calls a function from YY: we'll have TRtrace("YY","key","YYfunction",...)
    • => bsearch de YY in iy
    • => bsearch de key in ik
    • for every i
    • call[i] > 0 ?
    • end for
    • If exists ix fulfilling this and orders[iy][ik][ix] = 1
    • The trace is activated
    • See more details in:
    • - trace.dat: define trace keywords and acronyms
    • - orders_trace.dat: define what trace will be activated, where must be sent the traces (file, standard output...)

  • – Generalities This module allows to generate dinamic traces. It allows to define different trace levels or kinds, with different keywords:

    • Level start : ENTREE, ENTREE2, ENTREE3
    • Level end : SORTIE, SORTIE2, SORTIE3
    • A level defines hoy important is a funtion. For example, we may use it to set a different scope of a function:
    • Static function : NIVEAU2 (ENTREE2, SORTIE2)
    • Public function : NIVEAU1 (ENTREE, SORTIE)
    • or
    • Fonction allocation : NIVEAU3 (ENTREE3, SORTIE3)
    • Fonction initialisation : NIVEAU2 (ENTREE2, SORTIE2)
    • Fonction manipulation : NIVEAU1 (ENTREE, SORTIE)
    • Use rules:
    • 1) Tracing a level means necesarily the trace of lower levels
    • 2) The keywords DEBUG1, DEBUG2, DEBUG3 are not associated to the Level notion, thus a trace order of previously mentioned keywords will generate a trace. Thus if a trace order of this level is specified all the former keywords will be traced
    • The files ORDER_FILE and TRACE_FILE must be defined for this module:
      • Keywords declaration – MOTSCLES
      • all ENTREE SORTIE PARSING .....
    • ORDER_FILE contains trace ordes: A trace order is a triplet (modApp, modCou, Key)
    • Example :
    • trace GB DYN all
    • Will trace all the functions of the module DYN that are being called from module GB.
    • trace all DYN all
    • Will trace all the functions of the module DYN
    • trace all DYN DEBUG1
    • Will trace all the functions of the module DYN with keyword DEBUG1
    • etc – Be careful with the management of TRACE_FILE and ORDER_FILE files, a wrong manament may provoque a program crash.

Renvoie un par un les tokens trouves dans line.

Renvoie un par un les tokens trouves dans line Un token est une entite lexicale qui represente soit :

  • une declaration d acronymes (dec_acro)
  • une declaration de mot-cles (dec_key)
  • un debut d ordre de trace (trace)
  • un debut d ordre de redirection (output)
  • un debut d ordre d interruption (stop)
  • un debut d ordre de demarrage (start)
  • une fin de ligne (end_of_line)
  • un meta identifiant (all)
  • un ordre d indentation (indent)
  • un ordre de non-indentation (noindent)
  • un separateur de commentaire (comment)
  • un identifiant (string)

int TRtrace ( char *  mod,
char *  key,
char *  function,
char *  pattern,
  ... 
)

Declaration d une trace.

Declare une ou plusieurs traces (module,key). sachant que keys est une chaine de caracteres de une ou plusieurs cles separees par des espaces.

  • Exemple : On se place dans le module 'GC' au niveau de la fonction GCexec. on ajoute

    TRtrace("GC","entry","GCexec"," procCall = %s\n",procCall);

    puis, on decide d activer cette trace a partir d un programme de test. Dans ce programme , on ecrira :

    TRsetTrace("","GC","entry");

    A l'execution, cela donnera : trace GC/GCexec/entry/ >>>> procCall = TRNinsFX

    Si on a : TRtrace("GC","entry parsing","GCexec"," procCall = %s\n",procCall);

    On aura la trace si un des 2 mots cles au moins est actif: Lorsque plusieurs mots cles sont actifs, seule la trace correspondant au premier (de la gauche vers la droite) mot rencontre dans 'TRtrace' est activee.

modnom du module dans lequel on se trouve
keynom du mot cle associe a la trace
functionnom de la fonction dans laquelle on se trouve
patternformat de sortie du message de trace
...arguments a tracer, selon le format 'pattern'
Returns
O si OK, -1 si erreur de syntaxe
